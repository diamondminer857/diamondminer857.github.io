Projekt: Testy propustnosti sítě – LXC HTTP + Docker NFS

Předmět: Virtualizace 1 | Vypracoval: Maxmilián Babič

1. Úvod a cíl projektu

Cílem tohoto projektu bylo navrhnout a otestovat infrastrukturu založenou na dvou typech systémových kontejnerů – LXC/LXD a Docker – běžících na jednom hostitelském serveru s Ubuntu 24.04 LTS. Projekt zkoumá propustnost sítě mezi kontejnery ve dvou scénářích:

  • Scénář A (rozdělené služby): HTTP server (Apache) v LXC kontejneru servíruje soubory uložené na vzdáleném NFS úložišti v Docker kontejneru.
  • Scénář B (vše na jednom kontejneru): HTTP server i soubory jsou lokálně v jednom LXC kontejneru bez použití NFS.

Součástí projektu jsou také testy výkonu CPU a paměti pomocí nástroje sysbench, porovnání výkonu hostitelského systému a kontejneru, a popis možností škálování.

2. Architektura řešení

Celá infrastruktura běží na jednom hostitelském serveru. Kontejnery komunikují přes interní síťový bridge lxdbr0 na IPv6 síti fd42:909c:d903:ecc7::/64.

Scénář A – rozdělené služby

  • LXC kontejner http-server: Ubuntu 24.04, Apache 2.4.58, paměťový limit 1 GB. Servíruje soubory z NFS mountu (/var/www/html).
  • Docker kontejner nfs-server: Alpine Linux (itsthenetwork/nfs-server-alpine), exportuje adresář /nfs-share přes NFSv4.

Scénář B – vše na jednom kontejneru

  • LXC kontejner http-server: Stejný kontejner, ale soubory jsou uloženy lokálně bez NFS.
Síťové propojení: Komunikace probíhá přes IPv6 na interním bridge lxdbr0. Host má adresu fd42:909c:d903:ecc7::1, LXC kontejner fd42:909c:d903:ecc7:216:3eff:feee:7a19. Docker NFS server naslouchá na portu 2049 hostu.

Výpis běžících kontejnerů – LXC (http-server) a Docker (nfs-server).

4. Postup realizace

A. Instalace LXD a Docker na hostu

sudo apt install -y docker.io
sudo snap install lxd
sudo lxd init --minimal

B. Vytvoření LXC kontejneru s HTTP serverem

# Vytvoření kontejneru s paměťovým limitem 1 GB
lxc launch ubuntu:24.04 http-server -c limits.memory=1GB

# Instalace Apache uvnitř kontejneru
lxc exec http-server -- bash -c "apt update && apt install -y apache2"

# Povolení privilegovaného režimu pro NFS mount
lxc config set http-server security.privileged true
lxc config set http-server raw.apparmor "mount fstype=nfs*, mount fstype=rpc_pipefs,"
lxc restart http-server

# Override systemd služby Apache
lxc exec http-server -- bash -c "mkdir -p /etc/systemd/system/apache2.service.d && \
cat > /etc/systemd/system/apache2.service.d/override.conf << 'EOF'
[Service]
PrivateTmp=false
ProtectSystem=false
ProtectHome=false
NoNewPrivileges=false
EOF"
lxc exec http-server -- systemctl daemon-reload
lxc exec http-server -- systemctl start apache2

C. Spuštění Docker NFS serveru

# Příprava sdíleného adresáře
mkdir -p ~/nfs-data
echo "Hello from NFS Docker container" > ~/nfs-data/index.html
dd if=/dev/urandom of=~/nfs-data/testfile.bin bs=1M count=10

# Spuštění NFS kontejneru
docker run -d --name nfs-server \
  --privileged \
  -v ~/nfs-data:/nfs-share \
  -e SHARED_DIRECTORY=/nfs-share \
  -e PERMITTED="*" \
  -p 2049:2049 \
  itsthenetwork/nfs-server-alpine:latest

D. Propojení – NFS mount do LXC kontejneru

# Instalace NFS klienta
lxc exec http-server -- apt install -y nfs-common

# Mount NFS share přes IPv6
lxc exec http-server -- mount -t nfs4 -o nolock,vers=4.0 \
  [fd42:909c:d903:ecc7::1]:/ /var/www/html

# Ověření
curl -6 http://[fd42:909c:d903:ecc7:216:3eff:feee:7a19]/
# Výstup: Hello from NFS Docker container

5. Testy propustnosti a výkonu

A. Test propustnosti sítě – iperf3

Měření propustnosti síťového spojení mezi hostitelským systémem a LXC kontejnerem přes interní bridge lxdbr0. 

# Server v kontejneru
lxc exec http-server -- iperf3 -s -D

# Klient na hostu
iperf3 -c fd42:909c:d903:ecc7:216:3eff:feee:7a19 -t 10

Výstup testu propustnosti iperf3 – průměrná propustnost 25.8 Gbits/sec.

6. Sledování využití paměti

Dle zadání byl LXC kontejner omezen na maximálně 1 GB paměti. 

# Nastavený limit
lxc config get http-server limits.memory   # 1GB

# Využití paměti v kontejneru
lxc exec http-server -- free -m            # total 953, used 30

# Využití dle LXD
lxc info http-server | grep -i memory      # 174.27MiB

# Docker kontejner
docker stats nfs-server --no-stream        # 27.87MiB

Sledování využití paměti – LXC kontejner (limit 1 GB, využití 174 MB) a Docker NFS server (28 MB).

7. Možnosti škálování výkonu

LXC/LXD kontejnery

LXD umožňuje dynamicky měnit přidělené prostředky za běhu kontejneru:

# Změna limitu paměti
lxc config set http-server limits.memory 2GB

# Omezení počtu CPU jader
lxc config set http-server limits.cpu 4

# CPU priorita
lxc config set http-server limits.cpu.priority 10

Docker kontejnery

Docker podporuje dynamický update prostředků:

# Omezení paměti
docker update --memory 1g --memory-swap 1g nfs-server

# Omezení CPU
docker update --cpus 2 nfs-server

Horizontální škálování

Pro zvýšení propustnosti v produkčním prostředí je možné nasadit více instancí HTTP serveru za load balancer a sdílet NFS úložiště. Docker Compose nebo Kubernetes umožňují automatické škálování počtu replik služby na základě zátěže.

8. Závěr

Projekt úspěšně demonstroval propojení dvou typů systémových kontejnerů (LXC/LXD a Docker) s využitím síťového protokolu NFS pro sdílení souborů. Hlavní zjištění:

  • Síťová propustnost mezi kontejnery na stejném hostu dosahuje 25.8 Gbps (přes virtuální bridge).
  • HTTP výkon s NFS vs. lokálním úložištěm je téměř identický (~131 req/s) při nasazení na jednom hostu. V reálné síti by NFS přidávalo měřitelnou latenci.
  • CPU a paměťový výkon LXC kontejneru je srovnatelný s hostem (rozdíl <1 %), protože LXC sdílí kernel s hostem bez virtualizačního overheadu.
  • Paměťový limit 1 GB byl úspěšně nastaven a ověřen, kontejner využíval ~174 MB při běžném provozu.
Celkový závěr: LXC kontejnery poskytují výkon velmi blízký nativnímu hostu, zatímco Docker je ideální pro snadné nasazení izolovaných služeb. Kombinace obou technologií umožňuje flexibilní a výkonnou infrastrukturu. Skutečný dopad NFS na propustnost by se projevil až při oddělení služeb na fyzicky rozdílné servery.
